# 紫都业务系统操作流程手册

> 适用范围：紫都网页后台、销售微信小程序、Supabase 云端数据库  
> 版本：2026-07 上线前版

## 1. 系统之间的关系

网页后台和销售小程序是同一个 B2B 业务系统的两个操作端，不是两套独立数据。

- **网页后台**：适合电脑端下单、订单管理、客户管理、库存、批次、采购、财务、分析和系统管理。
- **销售小程序**：适合销售在手机端查价、下单、管客户、申请未收款发货、发起售后和发货。
- **Supabase**：两个端共用的云端数据库，订单、商品、价格、库存、客户、收付款和售后记录都从这里读写。
- **微信零售商城**：如果接入 2C 零售订单，会以“微信商城”来源进入同一订单与财务汇总，但它不是销售人员使用的 B2B 下单小程序。

同一笔订单不应该在网页和小程序重复建立。订单编号与“下单来源”用于识别它从哪个端创建。

## 2. 角色与职责

| 角色 | 主要操作 |
|---|---|
| 销售 | 客户、查价、下单、收款、发货、发起售后、申请未收款发货 |
| 仓库 | 备货、打印发货单、发货、退货验收、退货入库 |
| 财务 | 收款流水、退款、报表、销售业绩与提成 |
| 管理员 | 所有业务查看，人员角色，商品、价格、库存、采购，异常订单，未收款发货审批 |

原则：操作人使用自己账号，不共用密码；人员离职后由管理员立即停用账号。

## 3. 正常订单流程

1. 选择客户，确认客户类型、负责销售、联系方式和地址。
2. 选择原料或成品及规格，输入数量。库存不足时系统不创建订单。
3. 核对商品编号、规格、数量、单价、折扣、运费和最终金额。
4. 提交订单后，系统会在一次事务中创建订单并扣减库存。
5. 收款时必选：微信、支付宝、对公账户转账、对私银行账户转账。全额收款与订单确认在同一事务完成，普通客户进入待发货，展会/线下客户直接完成。
6. 订单进入待发货后，仓库先打印发货单备货核对，再填快递公司和单号发货。
7. 收货后将订单标记为已完成。

### 未收款但需发货

- 销售在订单详情的收款信息区选择“未收款，请发货”，填写原因。
- 管理员审核通过后才能发货。
- 管理员也可直接为任何订单批准，但原因仍必须记录。
- 未收款发货不代表已收款，财务报表仍显示未收金额。

## 4. 订单下错与取消

### A. 未收款、未发货

1. 订单里没有任何收款或退款流水时，可直接“取消订单”。
2. 取消与库存恢复在同一个数据库事务中完成；一项失败则全部不保存。
3. 取消单保留在订单列表供审计，不计入销售额、提成和正常订单统计。

### B. 已有收款、未发货

不得直接取消或删除。需要恢复商品库存时发起“退货退款”；不需要变动库存时发起“仅退款”。完成后订单保留，正数收款与负数退款在财务流水中对冲。

### C. 已发货

不能取消，只能走售后流程。

### D. 删除订单

“删除”只用于无收付款、无售后记录的重复单或误建单。删除后自动恢复库存，进入管理员删除订单库保留 30 天。已有收付款或售后记录时数据库会拒绝删除。

## 5. 退货退款流程

1. 销售或管理员在订单详情发起“退货退款”。
2. 按每个商品选择退货数量，可填备注。
3. 部分退货按订单原单价 × 退货数量自动计算；退回全部剩余商品时，系统自动按当前实收金额整单退，折扣和运费不会造成多退。
4. 仓库验货后选择“退回已入库”或“退回不入库”。
5. 财务选择退款方式并确认固定退款金额。
6. 系统同时写入负数付款流水、减少订单商品数量与金额、更新已收金额。
7. 退全部商品就是整单退，原订单保留为完整审计记录。

同一订单同一时间只能有一张待处理售后。仓库和财务重复点击时，数据库只允许第一次成功。

误建售后可由管理员在仓库尚未入库、财务尚未退款前取消。仓库已经处理后不得取消，必须继续完成财务处理，避免库存已增加但退款记录缺失。

## 6. 仅退款流程

1. 销售或管理员选择“仅退款”。
2. 输入退款金额并填写理由。
3. 不经过仓库，直接进入待财务处理。
4. 财务确认原申请金额后完成退款，系统记录负数流水并减少订单最终金额，库存不变。

## 7. 库存操作

### 原料

- 每个原料产品独立管理 kg 重量库存。
- ml 规格的可售数量由该产品自己的密度换算，不与其他精油共用库存。
- 密度在库存页右上角“密度换算”中查看和修改。
- 入库时优先走批次入库，记录批次号、供应商、入库日期、保质期和 GC-MS 编号。
- 出库先消耗历史无批次库存，再按有效批次入库日期从早到晚自动扣减；过期批次不会自动销售。
- 日常出库使用“自动 FIFO”；有指定批号或追溯要求时切换“指定批次”，系统同步扣减总库存与该批次余量。
- 过期批次不能作为正常出库批次，只能从批次追溯页按“报废”处理。

### 成品与包材

- 按每个 SKU 的瓶/个/箱数量独立管理。
- 采购到货应从采购单执行收货，系统同时更新已收数量、批次和库存。
- 盘点差异用“调整/盘点修正”，必须填原因，不直接改数据库。
- “库存全景”用于核对系统总库存、批次剩余、过期库存、无批次库存和下一个 FIFO 出库批次。
- “批次 / GC-MS 追溯”可直接对某一批次执行出库或报废，所有操作都进入批次流水与出入库记录。

## 8. 采购与入库

1. 建立供应商。
2. 建立采购单，选择原料或成品规格、采购数量和单位成本。
3. 发出采购后将状态设为已下单。
4. 到货时从采购单执行“收货/批次入库”，不重复在库存页再加一次。
5. 部分到货保持“部分收货”，全部到货后转为“已收货”。

“直接入库”只用于期初盘点、无采购单的历史库存或经管理员确认的特殊入库。

## 9. 财务核对

### 每日

1. 查看待收款、部分收款和未收款已发货订单。
2. 查看待财务处理售后。
3. 核对当日微信、支付宝、对公和对私银行流水。
4. 退款必须从售后工单完成，不在订单里手工改“已收金额”。

### 每月

1. 订单汇总：按下单日期统计应收、已收、未收。
2. 收付款流水：按实际收款/退款时间统计；退款为负数。
3. 商品明细：查产品编号、规格、数量、单价、折扣分摊和净销售额。
4. 销售提成：公司承担折扣不减销售提成基数；未点“公司承担”时默认由销售承担。
5. 导出 Excel 后留存月度副本。

“销售分析”使用有效订单最终金额，“收付款流水”使用实际入账日期，两者用途不同，不应要求每个时间段完全相等。

## 10. 上线前数据库更新

在 Supabase SQL Editor 按顺序运行：

1. `migration_v31_unpaid_shipping_approval.sql`
2. `migration_v32_no_backorder_atomic_orders.sql`
3. `migration_v33_admin_unpaid_shipping_override.sql`
4. `migration_v34_order_after_sales_integrity.sql`
5. `migration_v35_fifo_batch_inventory.sql`
6. `migration_v36_manual_batch_outbound.sql`
6. `verify_launch_integrity.sql`

`v34` 不清空产品、库存或订单，只增加原子事务和数据完整性保护；`v35` 增加 FIFO 批次扣减，`v36` 增加指定批次出库。运行结果中所有 `*_ready` 应为 `true`。

## 11. 安全与备份

Supabase 负责托管 Postgres 数据库，但“数据在 Supabase”不自动等于应用权限安全。上线前需要：

1. 将现有自定义手机号登录迁移到 Supabase Auth，让每个请求携带可验证的用户 JWT。
2. 删除业务表的 `Allow all ... USING (true) WITH CHECK (true)` 宽松策略。
3. 按管理员、销售、仓库、财务建立最小权限 RLS；销售只查自己客户/订单，仓库不查价格和财务。
4. 网页和小程序只放可公开的 publishable/anon key，永远不放 `service_role` 或 secret key。
5. Supabase 平台账号和 GitHub 开启 MFA，数据库开启 SSL Enforcement。
6. 检查当前套餐的自动备份保留时间；至少每月导出一份异地备份。
7. 每次大批量导入、价格更新或 SQL 迁移前，先确认最新备份可用。

完成 Auth + RLS 改造前，系统可以进行内部功能测试，但不应被定义为“已完成权限安全加固的正式生产系统”。

## 12. 故障处理

- 页面提示找不到 `zidu_*` 函数：检查是否按顺序运行最新 migration，等待约 10 秒刷新 schema cache 后重试。
- 退货无法提交：检查是否已收款、是否已有待处理售后、退货数量是否超出当前订单数量。
- 退款无法完成：检查是否选择有效退款方式，申请金额由系统固定。
- 取消被拒绝：订单已有收付款或已发货，应改走售后。
- 库存不一致：暂停继续出入库，保留截图和订单号，运行 `verify_launch_integrity.sql` 核对，不直接手改数据库。
